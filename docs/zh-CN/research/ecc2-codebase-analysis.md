# ECC2 代码库研究报告

**日期：** 2026-03-26
**主题：** `ecc-tui` v0.1.0 — 智能体 IDE 控制平面
**总行数：** 4,417 行，分布在 15 个 `.rs` 文件中

## 1. 架构概览

ECC2 是一个 Rust TUI 应用程序，用于编排 AI 编码智能体会话。它使用：

* **ratatui 0.29** + **crossterm 0.28** 用于终端 UI
* **rusqlite 0.32**（捆绑）用于本地状态持久化
* **tokio 1**（完整）用于异步运行时
* **clap 4**（派生）用于 CLI

### 模块分解

| 模块 | 行数 | 用途 |
|--------|------:|---------|
| `session/` | 1,974 | 会话生命周期、持久化、运行时、输出 |
| `tui/` | 1,613 | 仪表板、应用循环、自定义组件 |
| `observability/` | 409 | 工具调用风险评分和日志记录 |
| `config/` | 144 | 配置（TOML 文件） |
| `main.rs` | 142 | CLI 入口点 |
| `worktree/` | 99 | Git 工作树管理 |
| `comms/` | 36 | 智能体间消息传递（仅发送） |

### 关键架构模式

* **`session/runtime.rs` 中的 DbWriter 线程** — 一个专用的操作系统线程，用于通过 `mpsc::unbounded_channel` 从异步上下文进行 SQLite 写入，并带有一次性确认。这是解决“异步环境下的 SQLite”问题的简洁方案。
* **具有强制转换的会话状态机**：`Pending → {Running, Failed, Stopped}`、`Running → {Idle, Completed, Failed, Stopped}` 等。
* **用于会话输出的环形缓冲区** — 每个会话 `OUTPUT_BUFFER_LIMIT = 1000` 行，具有自动淘汰机制。
* **工具调用的风险评分** — 4 轴分析（基础工具风险、文件敏感性、影响范围、不可逆性），生成 0.0–1.0 的综合评分，并给出建议操作（允许/审查/需要确认/阻止）。

## 3. 已识别的差距

### 3.1 通信模块 — 仅发送无接收

`comms/mod.rs`（36 行）有 `send()`，但没有 `receive()`、`poll()`、`inbox()` 或 `subscribe()`。SQLite 中存在 `messages` 表，但没有代码从中读取。智能体间消息传递的故事只完成了一半。

**影响：** 智能体无法协调。定义了 `TaskHandoff`、`Query`、`Response` 和 `Conflict` 消息类型，但无法使用。

### 3.2 新建会话对话框 — 存根

`dashboard.rs:495` — `new_session()` 记录 `"New session dialog requested"` 但不执行任何操作。用户必须使用 CLI（`ecc start --task "..."`）来创建会话；TUI 仪表板无法创建。

### 3.3 单一智能体支持

`session/manager.rs` — `agent_program()` 仅支持 `"claude"`。CLI 接受 `--agent`，但任何非 `"claude"` 的值都会失败。不支持 codex、opencode 或自定义智能体。

### 3.4 配置 — 仅文件

`Config::load()` 仅读取 `~/.claude/ecc2.toml`。实现缺少环境变量覆盖（例如 `ECC_DB_PATH`、`ECC_WORKTREE_ROOT`）和用于配置的 CLI 标志。

### 3.5 遗留依赖候选：`git2`

`git2 = "0.20"` 仍然在 `Cargo.toml` 中声明，但 `worktree` 模块改为调用 `git` CLI。这使得 `git2` 成为一个强有力的移除候选，而不是已经完成的清理。

### 3.6 无指标聚合

`SessionMetrics` 跟踪每个会话的令牌数、成本、持续时间、工具调用次数、更改的文件数。但没有聚合视图：跨会话的总成本、平均持续时间、使用最多的工具等。仪表板中的指标窗格仅显示每个会话的详细信息。

### 3.7 守护进程 — 无健康报告

`session/daemon.rs` 运行一个无限循环检查会话超时。没有健康端点，没有日志轮换，没有 PID 文件，没有用于优雅关闭的信号处理。在守护进程模式下 `Ctrl+C` 会不干净地终止进程。

## 5. 安全观察

* **代码中没有秘密。** 配置从 TOML 文件读取，没有硬编码的凭据。
* **进程生成** 使用 `tokio::process::Command` 并显式设置 `Stdio::piped()` — 没有 shell 注入向量。
* **风险评分** 是一个强大的功能 — 捕获 `rm -rf`、`git push --force origin main`、对 `.env`/秘密文件的访问。
* **会话任务字符串没有输入清理。** 任务字符串直接传递给 `claude --print`。如果任务包含 shell 元字符，则可能被利用，具体取决于 `Command` 如何处理参数引用。目前是安全的（参数不会被 shell 解释），但值得审计。

## 7. 建议（按优先级排序）

### P0 — 快速见效

1. **为 `Config::load()` 添加环境变量支持** — `ECC_DB_PATH`、`ECC_WORKTREE_ROOT`、`ECC_DEFAULT_AGENT`。这是 CLI 工具的标准做法。

### P1 — 功能完善

2. **实现 `comms::receive()` / `comms::poll()`** — 从 `messages` 表中读取未读消息，可选地使用 `broadcast` 通道进行实时传递。将其连接到仪表板。
3. **在 TUI 中构建新建会话对话框** — 带有任务输入、智能体选择器、工作树切换的模式表单。应调用 `session::manager::create_session()`。
4. **添加聚合指标** — 总成本、平均会话持续时间、工具调用频率、每次会话成本。在指标窗格中显示。

### P2 — 健壮性

5. **扩展 `manager.rs`、`runtime.rs` 和 `daemon.rs` 的集成覆盖** — 仓库现在有了这些模块的基线测试，但仍需要围绕进程崩溃、超时和清理边缘情况的故障路径覆盖。
6. **为 `worktree/mod.rs` 和 `comms/mod.rs` 添加第一方测试** — 这些模块仍然未被覆盖，并且支持重要的编排功能。
7. **添加守护进程健康报告** — PID 文件、结构化日志记录、通过信号处理器优雅关闭。
8. **任务字符串安全审计** — 会话任务通过 `claude --print` 使用 `tokio::process::Command`。验证参数永远不会被 shell 解释。检查清单：确认 `Command` 参数用法、威胁模型元字符注入、输入验证/转义策略、原始输入日志记录以及自动化测试。如果调用代码发生变化，则重新审计。
9. **拆分 `dashboard.rs`** — 将会话窗格、输出窗格、指标窗格、日志窗格提取到 `tui/panes/` 下的单独文件中。

### P3 — 可扩展性

10. **多智能体支持** — 使 `agent_program()` 可插拔。添加 `codex`、`opencode`、`custom` 智能体类型。
11. **配置验证** — 验证风险阈值总和是否正确、预算值为正、路径存在。

## 8. 与 Ratatui 0.29 最佳实践对比

代码库很好地遵循了 ratatui 约定：

* 使用 `TableState` 进行有状态选择（正确模式）
* 为 `Widget` 实现自定义 `TokenMeter` 特性（符合惯例）
* 使用 `tick()` 方法进行定期状态同步（标准做法）
* 使用 `broadcast::channel` 处理实时输出事件（合适）

**细微偏差：**

* `Dashboard` 结构体直接持有 `StateStore`（SQLite 连接）。Ratatui 的最佳实践是将状态存储放在 `Arc<Mutex<>>` 后面，以允许后台更新。目前 TUI 独占拥有数据库，这阻碍了添加后台指标刷新任务。
* 渲染帮助覆盖层时未使用 `Clear` 组件 — 在某些终端上可能导致渲染伪影。
