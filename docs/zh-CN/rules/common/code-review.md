# 代码审查标准

## 目的

代码审查确保代码在合并前具备质量、安全性和可维护性。本规则定义了何时以及如何进行代码审查。

## 何时进行审查

**强制审查触发条件：**

* 编写或修改代码后
* 提交到共享分支之前
* 更改安全敏感代码时（认证、支付、用户数据）
* 进行架构变更时
* 合并拉取请求之前

**审查前要求：**

在请求审查前，请确保：

* 所有自动化检查（CI/CD）均已通过
* 合并冲突已解决
* 分支已更新至目标分支的最新状态

## 审查清单

在标记代码完成前：

* \[ ] 代码可读且命名良好
* \[ ] 函数职责单一（<50 行）
* \[ ] 文件内聚性高（<800 行）
* \[ ] 无深层嵌套（>4 层）
* \[ ] 错误已显式处理
* \[ ] 无硬编码的密钥或凭据
* \[ ] 无 console.log 或调试语句
* \[ ] 新功能已有测试
* \[ ] 测试覆盖率满足最低 80% 要求

## 安全审查触发条件

**遇到以下情况时，请立即停止并使用安全审查代理：**

* 认证或授权代码
* 用户输入处理
* 数据库查询
* 文件系统操作
* 外部 API 调用
* 加密操作
* 支付或金融代码

## 审查严重级别

| 级别 | 含义 | 处理措施 |
|-------|---------|--------|
| 严重 | 安全漏洞或数据丢失风险 | **阻止** - 必须在合并前修复 |
| 高 | 错误或重大质量问题 | **警告** - 应在合并前修复 |
| 中 | 可维护性问题 | **信息** - 建议修复 |
| 低 | 风格或次要建议 | **备注** - 可选 |

## 代理使用

使用以下代理进行代码审查：

| 代理 | 用途 |
|-------|---------|
| **code-reviewer** | 通用代码质量、模式、最佳实践 |
| **security-reviewer** | 安全漏洞、OWASP Top 10 |
| **typescript-reviewer** | TypeScript/JavaScript 特定问题 |
| **python-reviewer** | Python 特定问题 |
| **go-reviewer** | Go 特定问题 |
| **rust-reviewer** | Rust 特定问题 |

## 审查工作流

```
1. 运行 git diff 以了解变更内容
2. 首先检查安全清单
3. 审查代码质量清单
4. 运行相关测试
5. 验证覆盖率 >= 80%
6. 使用合适的代理进行详细审查
```

## 常见需捕获的问题

### 安全

* 硬编码凭据（API 密钥、密码、令牌）
* SQL 注入（查询中的字符串拼接）
* XSS 漏洞（未转义的用户输入）
* 路径遍历（未清理的文件路径）
* 缺少 CSRF 保护
* 认证绕过

### 代码质量

* 大型函数（>50 行）- 拆分为更小的函数
* 大型文件（>800 行）- 提取为模块
* 深层嵌套（>4 层）- 使用提前返回
* 缺少错误处理 - 显式处理错误
* 可变模式 - 优先使用不可变操作
* 缺少测试 - 增加测试覆盖率

### 性能

* N+1 查询 - 使用 JOIN 或批处理
* 缺少分页 - 为查询添加 LIMIT
* 无限制查询 - 添加约束条件
* 缺少缓存 - 缓存昂贵操作

## 批准标准

* **批准**：无严重或高级别问题
* **警告**：仅存在高级别问题（谨慎合并）
* **阻止**：发现严重问题
