# 编码风格

## 不可变性（关键）

始终创建新对象，切勿修改现有对象：

```
// 伪代码
错误：modify(original, field, value) → 原地修改 original
正确：update(original, field, value) → 返回带有更改的新副本
```

理由：不可变数据可以防止隐藏的副作用，使调试更容易，并实现安全的并发。

## 核心原则

### KISS（保持简单）

* 优先选择实际可行的最简单解决方案
* 避免过早优化
* 为清晰度而非巧妙性进行优化

### DRY（不要重复自己）

* 将重复的逻辑提取到共享函数或工具中
* 避免复制粘贴导致的实现偏差
* 当重复是真实存在而非推测时，才引入抽象

### YAGNI（你不需要它）

* 不要构建尚未需要的功能或抽象
* 避免推测性的通用性
* 从简单开始，然后在真正有压力时进行重构

## 文件组织

许多小文件 > 少数大文件：

* 高内聚，低耦合
* 典型 200-400 行，最多 800 行
* 从大型模块中提取工具函数
* 按功能/领域组织，而非按类型

## 错误处理

始终全面处理错误：

* 在每个层级都显式处理错误
* 在面向用户的代码中提供友好的错误信息
* 在服务器端记录详细的错误上下文
* 切勿静默吞没错误

## 输入验证

始终在系统边界进行验证：

* 在处理前验证所有用户输入
* 在可用时使用基于模式的验证
* 快速失败并提供清晰的错误信息
* 切勿信任外部数据（API 响应、用户输入、文件内容）

## 命名约定

* 变量和函数：`camelCase` 并使用描述性名称
* 布尔值：优先使用 `is`、`has`、`should` 或 `can` 前缀
* 接口、类型和组件：`PascalCase`
* 常量：`UPPER_SNAKE_CASE`
* 自定义钩子：`camelCase` 并带有 `use` 前缀

## 需避免的代码异味

### 深层嵌套

一旦逻辑开始堆叠，优先使用提前返回而非嵌套条件语句。

### 魔法数字

对有意义的阈值、延迟和限制使用命名常量。

### 长函数

将大型函数拆分为职责明确、重点突出的部分。

## 代码质量检查清单

在标记工作完成前：

* \[ ] 代码可读且命名良好
* \[ ] 函数短小（<50 行）
* \[ ] 文件重点突出（<800 行）
* \[ ] 无深层嵌套（>4 层）
* \[ ] 正确的错误处理
* \[ ] 无硬编码值（使用常量或配置）
* \[ ] 无数据修改（使用不可变模式）
