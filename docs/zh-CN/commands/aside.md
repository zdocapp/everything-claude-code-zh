# Aside 命令

在任务中途提问，获得即时、聚焦的答案——然后立即从暂停处继续。当前任务、文件和上下文绝不会被修改。

## 何时使用

* 当 Claude 正在工作时，你对某事感到好奇，并且不想失去工作势头
* 你需要快速解释 Claude 当前正在编辑的代码
* 你想就某个决定征求第二意见或澄清，而不想打乱当前任务
* 在 Claude 继续之前，你需要理解一个错误、概念或模式
* 你想询问与当前任务无关的事情，而无需开启新会话

## 用法

```
/aside <你的问题>
/aside 这个函数实际返回什么？
/aside 这个模式是线程安全的吗？
/aside 为什么这里使用 X 而不是 Y？
/aside foo() 和 bar() 有什么区别？
/aside 我们应该担心刚刚添加的 N+1 查询吗？
```

## 流程

### 步骤 1：冻结当前任务状态

在回答任何问题之前，请在心里记下：

* 当前活动任务是什么？（正在处理哪个文件、功能或问题）
* 在调用 `/aside` 时，进行到哪一步了？
* 接下来原本要发生什么？

在 aside 期间，请勿触摸、编辑、创建或删除任何文件。

### 步骤 2：直接回答问题

以最简洁但仍完整有用的形式回答问题。

* 先说答案，再说推理过程
* 保持简短——如果需要完整解释，请在任务结束后再深入
* 如果问题涉及当前正在处理的文件或代码，请精确引用（如果相关，包括文件路径和行号）
* 如果回答需要读取文件，请读取它——但只读不写

按以下格式回复：

```
ASIDE: [简要重述问题]

[你的回答在此]

— 返回任务: [一行描述正在进行的操作]
```

### 步骤 3：恢复主任务

在给出答案后，立即从暂停的确切点继续执行活动任务。除非 aside 的答案揭示了阻碍或需要重新考虑当前方法的原因（参见边缘情况），否则不要请求恢复许可。

***

## 边缘情况

**未提供问题（`/aside` 后面没有内容）：**
回复：

```
ASIDE: 未提供问题

您想了解什么？（提出您的问题，我会在不丢失当前任务上下文的情况下回答）

— 返回任务：[一行描述正在执行的操作]
```

**问题揭示了当前任务的潜在问题：**
在恢复之前清楚地标记出来：

```
ASIDE: [answer]

警告：注意：此答案指出当前方法存在[问题]。希望在继续之前解决此问题，还是按计划进行？
```

等待用户的决定后再恢复。

**问题实际上是任务重定向（而非附带问题）：**
如果问题暗示要改变正在构建的内容（例如，`/aside actually, let's use Redis instead`），请澄清：

```
ASIDE: 这听起来像是方向改变，而不仅仅是一个附带问题。
你想：
  (a) 仅将此作为信息回答，并保持当前计划
  (b) 暂停当前任务并改变方法
```

等待用户的回答——不要做假设。

**问题涉及当前打开的文件或代码：**
根据实时上下文回答。如果该文件在会话早期已被读取，请直接引用。如果尚未读取，现在读取它（只读）并在回答时附带文件:行号引用。

**无活动任务（调用 `/aside` 时没有进行中的任务）：**
仍使用标准包装器，以保持响应格式一致：

```
ASIDE: [简要重述问题]

[你的回答在此]

— 返回任务：无活动任务可恢复
```

**问题需要长篇回答：**
简洁地给出核心答案，然后提供：

```
这是简短版本。完成[当前任务]后需要更详细的解释吗？
```

**连续多个 `/aside` 问题：**
按顺序回答每一个问题。在回答完最后一个问题后，恢复主任务。不要在一连串的旁白中丢失任务状态。

**Aside 答案暗示需要进行代码更改：**
记下所需的更改，但不要在 aside 期间进行更改：

```
ASIDE: [答案]

值得修复：[需要更改的内容]。除非您想现在处理，否则我将在当前任务后标记此问题。
```

**问题含糊不清或过于模糊：**
提出一个澄清性问题——即获取回答问题所需信息的最简短问题。不要提出多个问题。

## 示例输出

```
User: /aside fetchWithRetry() 具体是做什么的？

ASIDE: fetchWithRetry() 是做什么的？

fetchWithRetry() (src/api/retry.ts:12) 会尝试请求最多 3 次，并采用指数退避策略（250ms → 500ms → 1s）。它仅在收到 5xx 响应和网络错误时重试 —— 4xx 错误被视为最终结果，不会重试。

— 回到任务：重构 src/middleware/auth.ts 中的身份验证中间件
```

```
User: /aside 是我们采用的方法线程安全的吗？

ASIDE: 当前的方法线程安全吗？

不 — src/cache/store.ts:34 中的共享缓存对象在没有锁定的情况下被修改。
在并发请求下，这是一个竞态条件。在单进程 Node.js 服务器中风险较低，但在使用工作线程或集群时，这将是一个真正的问题。

警告：注意：这可能会影响我们正在构建的功能。你想现在解决这个问题，还是继续并在后续修复？
```

***

## 注意事项

* 在 aside 期间切勿修改文件——仅限只读访问
* aside 是对话暂停，而非新任务——原始任务必须始终恢复
* 保持答案聚焦：目标是快速为用户扫清障碍，而非进行长篇大论
* 如果 aside 引发了更广泛的讨论，请先完成当前任务，除非 aside 揭示了阻碍
* 除非明确与任务结果相关，否则旁白内容不会保存到会话文件中
