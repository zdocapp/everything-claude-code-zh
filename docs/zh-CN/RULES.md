# 规则

## 必须始终

* 将领域任务委托给专门的智能体。
* 在实现前编写测试并验证关键路径。
* 验证输入并保持安全检查完整。
* 优先使用不可变更新而非修改共享状态。
* 遵循既有的仓库模式，而非发明新方法。
* 保持贡献内容聚焦、可审查且描述清晰。

## 严禁

* 在输出中包含敏感数据，如 API 密钥、令牌、密钥或绝对/系统文件路径。
* 提交未经测试的更改。
* 绕过安全检查或验证钩子。
* 在没有明确理由的情况下重复现有功能。
* 未检查相关测试套件就发布代码。

## 智能体格式

* 智能体位于 `agents/*.md`。
* 每个文件包含 YAML 前置元数据，其中包含 `name`、`description`、`tools` 和 `model`。
* 文件名为小写字母并使用连字符，且必须与智能体名称匹配。
* 描述必须清晰说明何时应调用该智能体。

## 技能格式

* 技能位于 `skills/<name>/SKILL.md`。
* 每个技能包含 YAML 前置元数据，其中包含 `name`、`description` 和 `origin`。
* 使用 `origin: ECC` 表示第一方技能，使用 `origin: community` 表示导入/社区技能。
* 技能正文应包含实用指南、经过测试的示例以及清晰的“何时使用”部分。

## 钩子格式

* 钩子使用匹配器驱动的 JSON 注册以及 Shell 或 Node 入口点。
* 匹配器应具体明确，而非宽泛的通用匹配。
* 仅当有意阻止行为时才退出 `1`；否则退出 `0`。
* 错误和信息消息应具有可操作性。

## 提交风格

* 使用约定式提交，例如 `feat(skills):`、`fix(hooks):` 或 `docs:`。
* 保持更改模块化，并在 PR 摘要中说明面向用户的影响。
