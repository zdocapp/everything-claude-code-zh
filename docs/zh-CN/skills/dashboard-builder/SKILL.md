# 仪表板构建器

当任务需要构建一个可供人们操作的仪表板时使用此技能。

目标不是“展示所有指标”。目标是回答：

* 它是否健康？
* 瓶颈在哪里？
* 发生了什么变化？
* 应该采取什么行动？

## 使用时机

* “构建一个 Kafka 监控仪表板”
* “为 Elasticsearch 创建一个 Grafana 仪表板”
* “为此服务制作一个 SigNoz 仪表板”
* “将这些指标列表转化为一个真正的运维仪表板”

## 指导原则

* 不要从视觉布局开始；从运维人员的问题开始
* 不要仅仅因为存在就包含所有可用指标
* 不要在没有结构的情况下混合健康、吞吐量和资源面板
* 不要交付没有标题、单位和合理阈值设置的面板

## 工作流程

### 1. 定义运维问题

围绕以下方面组织：

* 健康状态 / 可用性
* 延迟 / 性能
* 吞吐量 / 流量
* 饱和度 / 资源
* 服务特定风险

### 2. 研究目标平台架构

首先检查现有仪表板：

* JSON 结构
* 查询语言
* 变量
* 阈值样式
* 分区布局

### 3. 构建最小可用看板

推荐结构：

1. 概览
2. 性能
3. 资源
4. 服务特定分区

### 4. 剔除华而不实的面板

每个面板都应回答一个实际问题。如果不是，则移除它。

## 示例面板集

### Elasticsearch

* 集群健康状态
* 分片分配
* 搜索延迟
* 索引速率
* JVM 堆内存 / GC

### Kafka

* 代理数量
* 复制不足的分区
* 消息输入 / 输出
* 消费者延迟
* 磁盘和网络压力

### API 网关 / 入口

* 请求速率
* p50 / p95 / p99 延迟
* 错误率
* 上游健康状态
* 活跃连接数

## 质量检查清单

* \[ ] 有效的仪表板 JSON
* \[ ] 清晰的分区分组
* \[ ] 存在标题和单位
* \[ ] 阈值/状态颜色有意义
* \[ ] 存在用于常见筛选的变量
* \[ ] 默认时间范围和刷新间隔合理
* \[ ] 没有无运维价值的华而不实面板
