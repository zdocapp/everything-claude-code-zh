# 邮件操作

当实际任务是邮箱工作时使用：分类、起草、回复、发送，或证明邮件已进入“已发送”文件夹。

这不是通用的写作技能。它是围绕实际邮件界面的操作员工作流。

## 技能栈

在相关时，将这些 ECC 原生技能纳入工作流：

* `brand-voice` 在起草任何面向用户的内容之前
* `investor-outreach` 用于面向投资者、合作伙伴或赞助商的邮件
* `customer-billing-ops` 当邮件主题是账单/支持事件而非一般通信时
* `knowledge-ops` 当邮件或主题之后应被捕获到持久上下文中时
* `research-ops` 当回复依赖于新的外部事实时

## 使用时机

* 用户要求分类收件箱或归档低价值邮件
* 用户想要起草、回复或新建一封外发邮件
* 用户想知道某封邮件是否已发送
* 用户想要证明使用了哪个账户、主题或“已发送”条目

## 防护栏

* 除非用户明确要求实时发送，否则先起草
* 没有真实的“已发送”文件夹或客户端确认，切勿声称邮件已发送
* 不要随意切换发件人账户；选择与项目和收件人匹配的账户
* 清理时不要删除不确定的业务邮件
* 如果任务实际上是 DM 或 iMessage 工作，则转交给 `messages-ops`

## 工作流

### 1. 确定确切的界面

在行动之前，确定：

* 哪个邮箱账户
* 哪个主题或收件人
* 任务是分类、起草、回复还是发送
* 用户是仅需要草稿还是需要实时发送

### 2. 在撰写前阅读主题

如果是回复：

* 阅读现有主题
* 识别最后一次外发联系
* 识别任何承诺、截止日期或未答复的问题

如果是新建外发邮件：

* 确定关系亲密度
* 选择正确的渠道和发件人账户
* 在起草前调用 `brand-voice`

### 3. 起草，然后验证

对于仅起草工作：

* 生成最终文案
* 说明发件人、收件人、主题和目的

对于实时发送工作：

* 首先验证确切的最终正文
* 通过选定的邮件界面发送
* 确认邮件已进入“已发送”文件夹或等效的已发送副本存储

### 4. 报告确切状态

使用确切的状态词：

* 已起草
* 待批准
* 已发送
* 已阻止
* 等待验证

如果发送界面被阻止，请保留草稿并报告确切的阻止原因，而不是在未说明的情况下即兴使用第二种传输方式。

## 输出格式

```text
邮件界面
- 账户
- 会话 / 收件人
- 请求的操作

草稿
- 主题
- 正文

状态
- 已草拟 / 已发送 / 已阻止
- 发送证明（如适用）

下一步
- 发送
- 跟进
- 归档 / 移动
```

## 陷阱

* 未经已发送副本检查，切勿声称发送成功
* 不要忽略主题历史并撰写无上下文的回复
* 不要将邮箱工作与 DM 或短信工作流混为一谈
* 不要暴露秘密、认证详情或不必要的邮件元数据

## 验证

* 响应中指明了账户和主题或收件人
* 任何发送声明都包含“已发送”证明或明确的客户端确认
* 最终状态是以下之一：已起草 / 已发送 / 已阻止 / 等待验证
