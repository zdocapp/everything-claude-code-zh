# 终端操作

当用户需要进行实际的仓库操作时使用此功能：运行命令、检查 git 状态、调试 CI 或构建、进行小范围修复，并准确报告更改内容和验证结果。

此技能有意比一般的编码指导范围更窄。它是一种以证据为先的终端执行操作流程。

## 技能栈

在相关时，将这些 ECC 原生技能纳入工作流程：

* `verification-loop` 用于在更改后进行精确的验证步骤
* `tdd-workflow` 当正确的修复需要回归测试覆盖时
* `security-review` 当涉及密钥、认证或外部输入时
* `github-ops` 当任务依赖于 CI 运行、PR 状态或发布状态时
* `knowledge-ops` 当需要将已验证的结果捕获到持久的项目上下文中时

## 使用时机

* 用户提到“修复”、“调试”、“运行这个”、“检查仓库”或“推送”
* 任务依赖于命令输出、git 状态、测试结果或已验证的本地修复
* 答案必须区分本地更改、本地验证、已提交和已推送

## 防护措施

* 编辑前先检查
* 如果用户仅要求审计/审查，则保持只读模式
* 优先使用仓库本地脚本和辅助工具，而非临时编写的包装器
* 在验证命令重新运行之前，不要声称已修复
* 除非分支确实已推送到上游，否则不要声称已推送

## 工作流程

### 1. 确定工作范围

明确：

* 确切的仓库路径
* 分支
* 本地差异状态
* 请求的模式：
  * 检查
  * 修复
  * 验证
  * 推送

### 2. 首先读取故障点

在更改任何内容之前：

* 检查错误
* 检查文件或测试
* 检查 git 状态
* 在盲目重新读取之前，先使用任何已提供的日志或上下文

### 3. 保持修复范围小

一次解决一个主要故障：

* 首先使用最小且有效的验证命令
* 仅在本地故障解决后，才升级到更大的构建/测试通过
* 如果命令持续失败且错误特征相同，停止广泛重试并缩小范围

### 4. 报告确切的执行状态

使用精确的状态词：

* 已检查
* 本地已更改
* 本地已验证
* 已提交
* 已推送
* 已阻塞

## 输出格式

```text
表面信息
- 仓库
- 分支
- 请求模式

证据
- 失败的命令 / 差异 / 测试

行动
- 变更内容

状态
- 已检查 / 本地已修改 / 本地已验证 / 已提交 / 已推送 / 已阻塞
```

## 常见陷阱

* 当可以读取实时仓库状态时，不要基于过时的记忆工作
* 不要将小范围修复扩大为全仓库范围的改动
* 不要使用破坏性的 git 命令
* 不要忽略无关的本地工作

## 验证

* 响应中提及验证命令或测试的名称
* 与 git 相关的工作提及仓库路径和分支
* 任何推送声明都包含目标分支和确切结果
