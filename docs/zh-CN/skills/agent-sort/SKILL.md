# 代理排序

当仓库需要一个项目特定的 ECC 表面，而非默认的完整安装时，使用此技能。

目标不是猜测“感觉有用”。目标是根据实际代码库的证据对 ECC 组件进行分类。

## 何时使用

* 项目只需要 ECC 的一个子集，而完整安装过于嘈杂
* 仓库技术栈清晰，但没人想手动逐个筛选技能
* 团队想要一个基于 grep 证据而非个人意见的可重复安装决策
* 需要将始终加载的日常 workflow 表面与可搜索的库/参考表面分开
* 仓库已偏离到错误的语言、规则或钩子集，需要清理

## 不可协商的规则

* 使用当前仓库作为唯一真相来源，而非通用偏好
* 每个 DAILY 决策必须引用具体的仓库证据
* LIBRARY 不意味着“删除”；它意味着“保持可访问但不默认加载”
* 不要安装当前仓库无法使用的钩子、规则或脚本
* 优先使用 ECC 原生表面；不要引入第二个安装系统

## 输出

按顺序生成以下工件：

1. DAILY 清单
2. LIBRARY 清单
3. 安装计划
4. 验证报告
5. 可选的路由器 `skill-library`（如果项目需要）

## 分类模型

仅使用两个桶：

* `DAILY`
  * 应为此仓库的每个会话加载
  * 与仓库的语言、框架、workflow 或操作员表面高度匹配
* `LIBRARY`
  * 保留有用，但不值得默认加载
  * 应保持可通过搜索、路由器技能或选择性手动使用来访问

## 证据来源

在进行任何分类之前，使用仓库本地证据：

* 文件扩展名
* 包管理器和锁定文件
* 框架配置
* CI 和钩子配置
* 构建/测试脚本
* 导入和依赖清单
* 明确描述技术栈的仓库文档

有用的命令包括：

```bash
rg --files
rg -n "typescript|react|next|supabase|django|spring|flutter|swift"
cat package.json
cat pyproject.toml
cat Cargo.toml
cat pubspec.yaml
cat go.mod
```

## 并行审查阶段

如果有可用的并行子代理，将审查拆分为以下阶段：

1. 代理
   * 分类 `agents/*`
2. 技能
   * 分类 `skills/*`
3. 命令
   * 分类 `commands/*`
4. 规则
   * 分类 `rules/*`
5. 钩子和脚本
   * 分类钩子表面、MCP 健康检查、辅助脚本和操作系统兼容性
6. 额外项
   * 分类上下文、示例、MCP 配置、模板和指导文档

如果没有子代理可用，则按相同阶段顺序运行。

## 核心工作流

### 1. 读取仓库

在对任何内容进行分类之前，确定真实的技术栈：

* 使用的语言
* 使用的框架
* 主要包管理器
* 测试栈
* 代码检查/格式化栈
* 部署/运行时表面
* 已存在的操作员集成

### 2. 构建证据表

对于每个候选表面，记录：

* 组件路径
* 组件类型
* 建议的桶
* 仓库证据
* 简短理由

使用此格式：

```text
skills/frontend-patterns | skill | DAILY | 84 个 .tsx 文件，存在 next.config.ts | 核心前端技术栈
skills/django-patterns   | skill | LIBRARY | 无 .py 文件，无 pyproject.toml       | 在此仓库中未活跃
rules/typescript/*       | rules | DAILY | 存在 package.json + tsconfig.json      | 活跃的 TS 仓库
rules/python/*           | rules | LIBRARY | 零个 Python 源文件                   | 仅保持可访问
```

### 3. 决定 DAILY 与 LIBRARY

提升到 `DAILY` 当：

* 仓库明确使用匹配的技术栈
* 组件足够通用，有助于每个会话
* 仓库已依赖相应的运行时或 workflow

降级到 `LIBRARY` 当：

* 组件偏离技术栈
* 仓库以后可能需要它，但不是每天都需要
* 它增加了上下文开销而没有直接相关性

### 4. 构建安装计划

将分类转化为行动：

* DAILY 技能 -> 安装或保留在 `.claude/skills/`
* DAILY 命令 -> 仅在仍有用时保留为显式 shim
* DAILY 规则 -> 仅安装匹配的语言集
* DAILY 钩子/脚本 -> 仅保留兼容的
* LIBRARY 表面 -> 保持可通过搜索或 `skill-library` 访问

如果仓库已使用选择性安装，则更新该计划，而不是创建另一个系统。

### 5. 创建可选的库路由器

如果项目想要一个可搜索的库表面，创建：

* `.claude/skills/skill-library/SKILL.md`

该路由器应包含：

* 关于 DAILY 与 LIBRARY 的简短说明
* 分组的触发关键词
* 库引用所在位置

不要在路由器内复制每个技能的主体。

### 6. 验证结果

计划应用后，验证：

* 每个 DAILY 文件存在于预期位置
* 未遗留过时的语言规则处于活动状态
* 未安装不兼容的钩子
* 最终的安装确实与仓库技术栈匹配

返回一个包含以下内容的简洁报告：

* DAILY 计数
* LIBRARY 计数
* 已移除的过时表面
* 待解决的问题

如果下一步是交互式安装或修复，则交接给：

* `configure-ecc`

如果下一步是重叠清理或目录审查，则交接给：

* `skill-stocktake`

如果下一步是更广泛的上下文修剪，则交接给：

* `strategic-compact`

## 输出格式

按此顺序返回结果：

```text
STACK
- 语言/框架/运行时摘要

DAILY
- 常驻项目及证据

LIBRARY
- 可搜索/参考项目及证据

INSTALL PLAN
- 应安装、移除或路由的项目

VERIFICATION
- 已执行的检查与剩余缺口
```
