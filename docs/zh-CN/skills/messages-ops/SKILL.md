# 消息操作

当任务是实时消息检索时使用：iMessage、私信、近期一次性验证码，或在后续操作前检查对话线程。

这不是邮件工作。如果主要操作界面是邮箱，请使用 `email-ops`。

## 技能栈

在相关时将以下 ECC 原生技能纳入工作流：

* 当消息任务实际上是邮箱工作时，使用 `email-ops`
* 当私信线程属于外联网络工作时，使用 `connections-optimizer`
* 当实时线程应通知目标定位或热路径外联时，使用 `lead-intelligence`
* 当需要将线程内容捕获到持久化上下文中时，使用 `knowledge-ops`

## 使用时机

* 用户说“阅读我的消息”、“查看短信”、“查看私信”或“查找验证码”
* 任务依赖于实时线程或近期发送到本地消息界面的验证码
* 用户想要证明检查了哪个来源或线程

## 防护措施

* 首先确定来源：
  * 本地消息
  * X / 社交媒体私信
  * 另一个需要浏览器访问的消息界面
* 未指明来源前，不要声称已检查线程
* 如果存在已检查的辅助工具或标准路径，不要临时使用原始数据库访问
* 如果身份验证或多因素认证阻碍访问界面，请报告确切的阻碍信息

## 工作流程

### 1. 确定确切线程

在进行任何操作之前，先确定：

* 消息界面
* 发件人 / 收件人 / 服务
* 时间窗口
* 任务是检索、检查还是为回复做准备

### 2. 起草前先阅读

如果任务可能转变为外联后续操作：

* 阅读最新的接收消息
* 识别未完成事项
* 然后根据需要移交至正确的外联技能

### 3. 将验证码处理为聚焦检索任务

对于一次性验证码：

* 首先在最近的本地消息窗口中搜索
* 尽可能按服务或发件人缩小范围
* 一旦找到验证码或聚焦搜索已穷尽，即停止

### 4. 报告确切证据

返回：

* 使用的来源
* 尽可能提供线程或发件人信息
* 时间窗口
* 确切状态：
  * 已读
  * 已找到验证码
  * 受阻
  * 等待回复草稿

## 输出格式

```text
来源
- 消息表面
- 发送者 / 线程 / 服务

结果
- 消息摘要或代码
- 时间窗口

状态
- 已读 / 代码已找到 / 已阻止 / 等待回复草稿
```

## 常见陷阱

* 不要混淆邮箱工作和私信/短信工作
* 未指明来源前，不要声称已检索
* 当要求是查找近期验证码时，不要在广泛搜索上浪费时间
* 不要在不暴露阻碍的情况下，持续重试受阻的身份验证路径

## 验证

* 响应中指明了消息来源
* 响应包含了发件人、服务、线程或明确的阻碍信息
* 最终状态是明确且有限的
