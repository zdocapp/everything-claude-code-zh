# SEO

通过技术正确性、性能和内容相关性来提升搜索可见性，而非使用噱头。

## 何时使用

在以下情况使用此技能：

* 审核可抓取性、可索引性、规范链接或重定向
* 改进标题标签、元描述和标题结构
* 添加或验证结构化数据
* 改进核心网页指标
* 进行关键词研究并将关键词映射到URL
* 规划内部链接或站点地图/robots文件的更改

## 工作原理

### 原则

1. 在内容优化之前先解决技术障碍。
2. 一个页面应有一个明确的主要搜索意图。
3. 优先考虑长期质量信号，而非操纵性模式。
4. 移动优先的假设很重要，因为索引是移动优先的。
5. 建议应针对具体页面且可实施。

### 技术SEO检查清单

#### 可抓取性

* `robots.txt` 应允许重要页面并阻止低价值页面
* 不应有任何重要页面被无意中 `noindex`
* 重要页面应在较浅的点击深度内可达
* 避免超过两次跳转的重定向链
* 规范标签应自洽且无循环

#### 可索引性

* 首选URL格式应保持一致
* 多语言页面如需使用，应具备正确的hreflang
* 站点地图应反映预期的公开页面
* 没有重复URL在缺乏规范控制的情况下相互竞争

#### 性能

* LCP < 2.5秒
* INP < 200毫秒
* CLS < 0.1
* 常见修复方法：预加载关键资源、减少渲染阻塞工作、预留布局空间、精简繁重的JS

#### 结构化数据

* 主页：酌情使用组织或企业架构
* 编辑页面：`Article` / `BlogPosting`
* 产品页面：`Product` 和 `Offer`
* 内容页面：`BreadcrumbList`
* 问答部分：仅当内容完全匹配时使用 `FAQPage`

### 页面规则

#### 标题标签

* 目标长度约为50-60个字符
* 将主要关键词或概念放在靠前位置
* 使标题对人类可读，而非为机器人堆砌

#### 元描述

* 目标长度约为120-160个字符
* 如实描述页面
* 自然地包含主要主题

#### 标题结构

* 一个清晰的 `H1`
* `H2` 和 `H3` 应反映实际的内容层次结构
* 不要仅为了视觉样式而跳过结构

### 关键词映射

1. 定义搜索意图
2. 收集现实的关键词变体
3. 根据意图匹配度、潜在价值和竞争度进行优先级排序
4. 将一个主要关键词/主题映射到一个URL
5. 检测并避免内部竞争

### 内部链接

* 从强页面链接到你希望排名的页面
* 使用描述性锚文本
* 在可能的情况下，避免使用通用锚文本，而选择更具体的
* 从新页面回填链接到相关的现有页面

## 示例

### 标题公式

```text
Primary Topic - Specific Modifier | Brand
```

### 元描述公式

```text
行动 + 主题 + 价值主张 + 一个支持性细节
```

### JSON-LD示例

```json
{
  "@context": "https://schema.org",
  "@type": "Article",
  "headline": "Page Title Here",
  "author": {
    "@type": "Person",
    "name": "Author Name"
  },
  "publisher": {
    "@type": "Organization",
    "name": "Brand Name"
  }
}
```

### 审计输出格式

```text
[HIGH] 产品页面上存在重复的标题标签
位置：src/routes/products/[slug].tsx
问题：动态标题会回退到相同的默认字符串，这削弱了相关性并产生了重复信号。
修复：使用产品名称和主要类别为每个产品生成唯一的标题。
```

## 反面模式

| 反面模式 | 修复方法 |
| --- | --- |
| 关键词堆砌 | 首先为用户写作 |
| 内容单薄的近似重复页面 | 合并或区分它们 |
| 为实际不存在的内容添加架构 | 使架构与现实匹配 |
| 未检查实际页面就给出内容建议 | 先阅读真实页面 |
| 泛泛的“改进SEO”输出 | 将每条建议与具体页面或资源关联 |
