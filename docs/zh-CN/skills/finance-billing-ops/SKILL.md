# 财务账单运营

当用户想要了解资金、定价、退款、团队席位逻辑，或者产品实际行为是否与网站和销售文案所暗示的一致时，请使用此技能。

这比 `customer-billing-ops` 更广泛。该技能用于客户补救。此技能用于运营真相：收入状态、定价决策、团队账单以及代码支持的账单行为。

## 技能栈

在相关时，将这些 ECC 原生技能纳入工作流程：

* `customer-billing-ops` 用于特定客户的补救和跟进
* `research-ops` 当竞争对手定价或当前市场证据重要时
* `market-research` 当答案应以定价建议结束时
* `github-ops` 当账单真相取决于兄弟仓库中的代码、待办事项或发布状态时
* `verification-loop` 当答案取决于证明结账、席位处理或授权行为时

## 使用时机

* 用户询问 Stripe 销售额、退款、月度经常性收入或近期客户活动
* 用户询问团队账单、按席位计费或配额叠加在代码中是否真实存在
* 用户需要竞争对手定价比较或定价模型基准
* 问题混合了收入事实与产品实现真相

## 防护栏

* 区分实时数据与保存的快照
* 分开：
  * 收入事实
  * 客户影响
  * 代码支持的产品真相
  * 建议
* 除非实际的授权路径强制执行，否则不要说"按席位"
* 不要假设重复订阅意味着重复价值

## 工作流程

### 1. 从最新的账单证据开始

优先使用实时账单数据。如果数据不是实时的，请明确说明快照时间戳。

规范化视图：

* 付费销售额
* 有效订阅
* 失败或未完成的结账
* 退款
* 争议
* 重复订阅

### 2. 将客户事件与产品真相分开

如果问题是针对特定客户的，请先分类：

* 重复结账
* 真实的团队意图
* 自助服务控制功能损坏
* 未满足的产品价值
* 支付失败或设置未完成

然后将其与更广泛的产品问题分开：

* 团队账单是否真的存在？
* 席位是否实际被计数？
* 结账数量是否改变授权？
* 网站是否夸大了当前行为？

### 3. 检查代码支持的账单行为

如果答案取决于实现真相，请检查代码路径：

* 结账
* 定价页面
* 授权计算
* 席位或配额处理
* 安装与用户使用逻辑
* 账单门户或自助服务管理支持

### 4. 以决策和产品差距结束

报告：

* 销售快照
* 问题诊断
* 产品真相
* 建议的运营操作
* 产品或待办事项差距

## 输出格式

```text
快照
- 时间戳
- 收入 / 订阅数 / 异常情况

客户影响
- 受影响对象
- 事件详情

产品实情
- 代码实际功能
- 网站或销售文案宣传内容

决策
- 退款 / 保留 / 转化 / 无操作

产品差距
- 待构建或修复的具体跟进事项
```

## 陷阱

* 不要将失败的尝试与净收入混为一谈
* 不要仅从营销语言推断团队账单
* 当有当前证据可用时，不要凭记忆比较竞争对手定价
* 不要从诊断直接跳到退款，而不对问题进行分类

## 验证

* 答案包含实时数据声明或快照时间戳
* 产品真相声明有代码支持
* 客户影响和更广泛的定价/产品结论被清晰地分开
