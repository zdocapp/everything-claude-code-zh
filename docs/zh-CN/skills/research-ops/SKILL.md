# 研究运营

当用户要求研究当前事物、比较选项、丰富人员或公司信息，或将重复查找转变为监控工作流时，使用此功能。

这是围绕仓库研究栈的运算符包装器。它不是 `deep-research`、`exa-search` 或 `market-research` 的替代品；它会告诉你何时以及如何一起使用它们。

## 技能栈

在相关时将以下 ECC 原生技能拉入工作流：

* `exa-search` 用于快速当前网络发现
* `deep-research` 用于多源综合并附带引用
* `market-research` 当最终结果应为推荐或排名决策时使用
* `lead-intelligence` 当任务是人员/公司定位而非通用研究时使用
* `knowledge-ops` 当结果之后应存储在持久上下文中时使用

## 使用时机

* 用户提到“研究”、“查找”、“比较”、“我应该联系谁”或“最新情况是什么”
* 答案取决于当前的公开信息
* 用户已提供证据并希望将其纳入新的推荐中
* 任务可能重复发生，应转变为监控而非一次性查找

## 防护栏

* 当新鲜搜索成本低廉时，不要根据过时的记忆回答当前问题
* 区分：
  * 已获取的事实
  * 用户提供的证据
  * 推断
  * 推荐
* 如果答案已存在于本地代码或文档中，不要启动重量级研究流程

## 工作流

### 1. 从用户已提供的内容开始

将任何提供的材料规范化为：

* 已证实的事实
* 需要验证的内容
* 开放性问题

如果用户已构建部分模型，不要从零重新开始分析。

### 2. 对请求进行分类

在搜索前选择正确的路径：

* 快速事实性答案
* 比较或决策备忘录
* 线索/信息丰富流程
* 重复监控候选

### 3. 首先采用最轻量但有用的证据路径

* 使用 `exa-search` 进行快速发现
* 当综合或多源很重要时，升级到 `deep-research`
* 当结果应以推荐结束时，使用 `market-research`
* 当实际请求是目标排名或暖路径发现时，移交到 `lead-intelligence`

### 4. 报告时明确证据边界

对于重要声明，说明它们是：

* 已获取的事实
* 用户提供的上下文
* 推断
* 推荐

对新鲜度敏感的答案应包含具体日期。

### 5. 决定任务是否应保持手动

如果用户可能重复询问相同的研究问题，请明确说明，并推荐监控或工作流层，而不是永远重复相同的手动搜索。

## 输出格式

```text
问题类型
- 事实性 / 对比性 / 补充性 / 监测性

证据
- 有来源的事实
- 用户提供的上下文

推理
- 从证据中得出的结论

建议
- 答案或下一步行动
- 是否应将其设为监测项
```

## 陷阱

* 不要将推断混入已获取的事实而不加标注
* 不要忽略用户提供的证据
* 不要对本地仓库上下文可以回答的问题使用重量级研究路径
* 不要在没有日期的情况下给出对新鲜度敏感的答案

## 验证

* 重要声明按证据类型标注
* 对新鲜度敏感的输出包含日期
* 最终推荐与实际使用的研究模式相符
