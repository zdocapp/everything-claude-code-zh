# 项目流程操作

此技能将分散的 GitHub 议题、PR 和 Linear 任务转化为一个执行流程。

当问题在于协调而非编码时使用它。

## 何时使用

* 处理积压的未处理 PR 或议题
* 决定哪些应归属 Linear，哪些应仅保留在 GitHub
* 将活跃的 GitHub 工作链接到内部执行通道
* 将 PR 分类为合并、移植/重构、关闭或搁置
* 审核审查评论、CI 失败或陈旧议题是否阻碍执行

## 操作模式

* **GitHub** 是公开和社区的真相来源
* **Linear** 是内部活跃计划工作的执行真相来源
* 并非每个 GitHub 议题都需要一个 Linear 议题
* 仅当工作符合以下条件时创建或更新 Linear：
  * 活跃
  * 已委派
  * 已计划
  * 跨职能
  * 足够重要以在内部跟踪

## 核心工作流

### 1. 首先阅读公开表面

收集：

* GitHub 议题或 PR 状态
* 作者和分支状态
* 审查评论
* CI 状态
* 关联议题

### 2. 对工作进行分类

每个项目应最终处于以下状态之一：

| 状态 | 含义 |
|-------|---------|
| 合并 | 自包含、符合政策、就绪 |
| 移植/重构 | 有用的想法，但应在 ECC 内手动重新落地 |
| 关闭 | 方向错误、陈旧、不安全或重复 |
| 搁置 | 可能有用，但当前未计划 |

### 3. 决定是否需要 Linear

仅在以下情况下创建或更新 Linear：

* 执行已积极计划
* 涉及多个仓库或工作流
* 工作需要内部所有权或排序
* 议题是更大项目通道的一部分

不要机械地镜像所有内容。

### 4. 保持两个系统一致

当工作活跃时：

* GitHub 议题/PR 应公开说明正在发生什么
* Linear 应在内部跟踪负责人、优先级和执行通道

当工作发布或被拒绝时：

* 将公开决议发布回 GitHub
* 相应标记 Linear 任务

## 审查规则

* 切勿仅凭标题、摘要或信任进行合并；使用完整差异
* 外部来源的功能，当它们有价值但不自包含时，应在 ECC 内重构
* CI 红色意味着分类并修复或阻止；不要假装它已准备好合并
* 如果真正的阻碍是产品方向，请明确指出，而不是隐藏在工具背后

## 输出格式

返回：

```text
公开状态
- issue / PR 状态
- CI / 审查状态

分类
- 合并 / 移植重建 / 关闭 / 搁置
- 一段理由说明

线性操作
- 创建 / 更新 / 无需 Linear 事项
- 项目 / 泳道（如适用）

下一步操作
- 确切的下一步行动
```

## 良好用例

* "审核未处理的 PR 积压，并告诉我哪些应合并 vs 重构"
* "将 GitHub 议题映射到我们的 ECC 1.x 和 ECC 2.0 项目通道"
* "检查这是否需要 Linear 议题，还是应仅保留在 GitHub"
