# 统一通知运维

当真正的问题不是缺少通知，而是通知系统碎片化时，请使用此技能。

这项工作的目标是将分散的事件转化为一个统一的运维界面，具备：

* 清晰的严重性
* 清晰的负责人
* 清晰的路由
* 清晰的后续操作

## 何时使用

* 用户希望跨 GitHub、Linear、本地钩子、桌面提醒、聊天或电子邮件建立一个统一的通知通道
* CI 失败、代码审查请求、问题更新和运维事件正分散到达不同的地方
* 当前设置产生了噪音而非行动
* 用户希望将重叠的通知分支或待办事项提案整合到一个 ECC 原生的通道中
* 工作空间已有钩子、MCP 或连接的工具，但缺乏一致的通知策略

## 首选界面

从已存在的部分开始：

* GitHub 问题、PR、审查、评论和 CI
* Linear 问题/项目动态
* 本地钩子事件和会话生命周期信号
* 桌面通知原语
* 已连接的电子邮件/聊天界面（当它们确实存在时）

优先采用 ECC 原生编排，而非建议用户采用单独的通知产品。

## 不可协商的规则

* 绝不暴露令牌、密钥、Webhook 密钥或内部标识符
* 分离：
  * 事件源
  * 严重性
  * 路由通道
  * 运维操作
* 当中断成本不明确时，默认优先采用摘要模式
* 不要将每个事件都广播到每个通道
* 如果真正的解决方案是更好的问题分类、钩子策略或项目流程，请明确指出

## 事件流水线

将此通道视为：

1. **捕获** 事件
2. **分类** 紧急程度和负责人
3. **路由** 到正确的通道
4. **合并** 重复项和低价值变更
5. **附加** 下一个运维操作

目标是更少、更好的通知。

## 默认严重性模型

| 类别 | 示例 | 默认处理方式 |
| --- | --- | --- |
| 严重 | 默认分支 CI 中断、安全问题、发布受阻、部署失败 | 立即中断 |
| 高 | 审查请求、PR 失败、负责人阻塞的交接 | 当日提醒 |
| 中 | 问题状态变更、重要评论、待办事项动态 | 摘要或排队 |
| 低 | 重复的成功、常规变更、冗余的生命周期标记 | 抑制或折叠 |

如果工作空间没有严重性模型，在提议自动化之前先建立一个。

## 工作流程

### 1. 盘点当前界面

列出：

* 事件源
* 当前通道
* 发出警报的现有钩子/脚本
* 同一事件的重复路径
* 重要事项未被呈现的静默失败情况

指出 ECC 已拥有的部分。

### 2. 决定哪些值得中断

对于每个事件族，回答：

* 谁需要知道？
* 他们需要多快知道？
* 这应该中断、批量处理还是仅记录？

使用以下默认值：

* 对于发布、CI、安全和负责人阻塞的事件进行中断
* 对于中等信号更新采用摘要
* 对于遥测和低价值生命周期标记仅记录

### 3. 在添加通道前合并重复项

寻找：

* 同一 PR 事件出现在 GitHub、Linear 和本地日志中
* 针对同一失败的重复钩子通知
* 应被总结而非原始转发的评论或状态变更
* 彼此重复但未提供更好操作路径的通道

优先选择：

* 一个规范的摘要
* 一个负责人
* 一个主通道
* 一个备用路径

### 4. 设计 ECC 原生工作流

对于每个真正的通知需求，定义：

* **来源**
* **门控**
* **形式**：即时警报、摘要、队列或仅仪表板
* **通道**
* **操作**

如果 ECC 已有原语，优先选择：

* 用于运维分类的技能
* 用于自动发出/执行的钩子
* 用于委托分类的代理
* 仅当确实缺少桥梁时才使用 MCP/连接器

### 5. 返回一个偏向行动的设计

以以下内容结束：

* 保留什么
* 抑制什么
* 合并什么
* ECC 接下来应该包装什么

## 输出格式

```text
当前表面
- 来源
- 渠道
- 重复项
- 缺口

事件模型
- 关键
- 高
- 中
- 低

路由计划
- 来源 -> 渠道
- 原因
- 操作员所有者

整合
- 抑制
- 合并
- 规范摘要

下一步 ECC 行动
- 技能 / 钩子 / 代理 / MCP
- 构建下一步的确切工作流程
```

## 推荐规则

* 优先选择一个强大的通道，而非多个弱小的通道
* 对于中低价值信号更新，优先采用摘要
* 当信号应自动发出时，优先采用钩子
* 当工作是分类、路由和审查优先决策时，优先采用运维技能
* 当根本原因是待办事项/PR 协调而非警报时，优先采用 `project-flow-ops`
* 当用户首先需要源清单时，优先采用 `workspace-surface-audit`
* 如果桌面通知足够，不要发明不必要的外部桥梁

## 良好用例

* "我们有 GitHub、Linear 和本地钩子警报，但没有统一的运维流程"
* "我们的 CI 失败通知很嘈杂，人们忽略了它们"
* "我希望在 Claude、OpenCode 和 Codex 界面间有一个统一的通知策略"
* "弄清楚哪些应该中断，哪些应该进入摘要"
* "将重叠的通知 PR 想法合并到一个规范的 ECC 通道中"
