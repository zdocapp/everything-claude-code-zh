# 安全赏金猎人

当目标是进行实际漏洞发现以进行负责任披露或赏金提交时使用，而非进行宽泛的最佳实践审查。

## 何时使用

* 扫描代码库以寻找可利用的漏洞
* 准备 Huntr、HackerOne 或类似赏金平台的提交
* 进行筛选，关注的问题是“这实际上能获得赏金吗？”而不是“这在理论上不安全吗？”

## 工作原理

偏向于远程可访问、用户可控的攻击路径，并丢弃那些平台通常会以信息性不足或超出范围为由拒绝的模式。

## 范围内的模式

这些是通常具有实际意义的漏洞类型：

| 模式 | CWE | 典型影响 |
| --- | --- | --- |
| 通过用户可控 URL 的 SSRF | CWE-918 | 内部网络访问、云元数据窃取 |
| 中间件或 API 防护中的认证绕过 | CWE-287 | 未经授权的账户或数据访问 |
| 远程反序列化或上传到 RCE 的路径 | CWE-502 | 代码执行 |
| 可访问端点中的 SQL 注入 | CWE-89 | 数据泄露、认证绕过、数据破坏 |
| 请求处理器中的命令注入 | CWE-78 | 代码执行 |
| 文件服务路径中的路径遍历 | CWE-22 | 任意文件读取或写入 |
| 自动触发的 XSS | CWE-79 | 会话窃取、管理员账户被攻陷 |

## 跳过这些

除非项目另有说明，否则这些通常是低价值或超出赏金范围的：

* 仅限本地的 `pickle.loads`、`torch.load` 或等效情况，且没有远程路径
* 仅在 CLI 工具中的 `eval()` 或 `exec()`
* 完全硬编码命令上的 `shell=True`
* 单独存在的缺少安全标头问题
* 没有实际利用影响的通用速率限制投诉
* 需要受害者手动粘贴代码的 Self-XSS
* 不属于目标项目范围的 CI/CD 注入
* 演示、示例或仅用于测试的代码

## 工作流程

1. 首先检查范围：项目规则、SECURITY.md、披露渠道和排除项。
2. 寻找真实的入口点：HTTP 处理器、上传功能、后台作业、Webhook、解析器和集成端点。
3. 在有用时运行静态分析工具，但仅将其视为筛选输入。
4. 从头到尾阅读真实的代码路径。
5. 证明用户控制能够到达一个有意义的接收点。
6. 用尽可能小的安全 PoC 确认可利用性和影响。
7. 在起草报告前检查重复项。

## 示例筛选循环

```bash
semgrep --config=auto --severity=ERROR --severity=WARNING --json
```

然后手动过滤：

* 丢弃测试、演示、固件、供应商代码
* 丢弃仅限本地或不可访问的路径
* 仅保留具有明确网络或用户可控路径的发现

## 报告结构

```markdown
## 描述
[漏洞是什么以及为何重要]

## 漏洞代码
[文件路径、行号范围及一小段代码片段]

## 概念验证
[最小可复现的请求或脚本]

## 影响
[攻击者能实现什么]

## 受影响版本
[测试的版本、提交或部署目标]
```

## 质量检查

提交前确认：

* 代码路径可以从真实用户或网络边界访问
* 输入确实是用户可控的
* 接收点有意义且可利用
* PoC 有效
* 该问题尚未被安全公告、CVE 或开放工单覆盖
* 目标确实在赏金项目的范围内
