# 产品透镜 —— 构建之前先思考

此流程负责产品诊断，而非撰写可直接实施的规范。

如果用户需要一份持久的 PRD 到 SRS 或能力契约工件，请转交给 `product-capability`。

## 何时使用

* 在启动任何功能之前 —— 验证“为什么”
* 每周产品评审 —— 我们是否在构建正确的东西？
* 在功能之间难以抉择时
* 发布之前 —— 对用户旅程进行合理性检查
* 在将模糊想法转化为产品简报，并开始工程规划之前

## 工作原理

### 模式 1：产品诊断

类似 YC 办公时间，但自动化。提出尖锐问题：

```
1. 这是为谁准备的？（具体的人，而非泛指“开发者”）
2. 痛点是什么？（量化：频率、严重程度、他们目前如何应对？）
3. 为什么是现在？（是什么变化使得这成为可能/必要？）
4. 10星级版本是什么？（如果资金/时间不受限）
5. MVP（最小可行产品）是什么？（能验证核心假设的最小可行方案）
6. 反目标是什么？（明确声明不构建什么？）
7. 如何衡量它是否有效？（具体指标，而非感觉）
```

输出：一份包含答案、风险和“执行/不执行”建议的 `PRODUCT-BRIEF.md`。

如果结果是“是的，构建这个”，则下一流程是 `product-capability`，而非更多创始人式的空谈。

### 模式 2：创始人评审

通过创始人视角评审你当前的项目：

```
1. 阅读 README、CLAUDE.md、package.json、最近的提交记录
2. 推断：这个项目试图成为什么？
3. 评分：产品-市场匹配信号（0-10分）
   - 使用增长轨迹
   - 留存指标（重复贡献者、回访用户）
   - 收入信号（定价页面、计费代码、Stripe集成）
   - 竞争护城河（哪些方面难以复制？）
4. 识别：能让这个项目增长10倍的那一件事
5. 标记：你正在构建但无关紧要的事情
```

### 模式 3：用户旅程审计

映射真实的用户体验：

```
1. 以新用户身份克隆/安装产品
2. 记录每一个摩擦点（令人困惑的步骤、错误、缺失的文档）
3. 为每个步骤计时
4. 与竞争对手的入门流程进行比较
5. 评分：价值实现时间（用户需要多久才能获得首次成功？）
6. 建议：优化入门流程的前3项改进措施
```

### 模式 4：功能优先级排序

当你有 10 个想法但需要选择 2 个时：

```
1. 列出所有候选功能
2. 为每个功能评分：影响度 (1-5) × 置信度 (1-5) ÷ 工作量 (1-5)
3. 按 ICE 分数排序
4. 应用约束条件：可用时间、团队规模、依赖关系
5. 输出：带有理由的优先级路线图
```

## 输出

所有模式都输出可操作的文档，而非长篇大论。每个建议都有具体的后续步骤。

## 集成

可与以下流程配合使用：

* `/browser-qa` 用于验证用户旅程审计结果
* `/design-system audit` 用于视觉美化评估
* `/canary-watch` 用于发布后监控
* `product-capability` 当产品简报需要转变为可实施的能力计划时
