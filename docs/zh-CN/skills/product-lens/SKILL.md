# 产品透镜 — 三思而后行

## 何时使用

* 在开始任何功能之前 — 验证“为什么”
* 每周产品评审 — 我们是否在做正确的事？
* 当在功能之间难以抉择时
* 发布前 — 对用户旅程进行合理性检查
* 当将一个模糊的想法转化为规格说明时

## 工作原理

### 模式 1：产品诊断

类似 YC 办公时间，但已自动化。提出尖锐问题：

```
1. 这是为谁准备的？（具体的人，而不是泛指“开发者”）
2. 痛点是什么？（量化：频率、严重程度、他们目前如何应对？）
3. 为什么是现在？（是什么变化使得这成为可能/必要？）
4. 10星级版本是什么？（如果资金/时间不受限制）
5. MVP是什么？（能证明核心理念的最小可行产品）
6. 反目标是什么？（明确声明你不会构建什么？）
7. 你如何知道它有效？（可量化的指标，而非感觉）
```

输出：一份包含答案、风险以及执行/放弃建议的 `PRODUCT-BRIEF.md`。

### 模式 2：创始人评审

通过创始人视角评审你当前的项目：

```
1. 阅读 README、CLAUDE.md、package.json、最近的提交记录
2. 推断：这个项目试图成为什么？
3. 评分：产品-市场匹配信号（0-10分）
   - 使用增长轨迹
   - 留存指标（重复贡献者、回访用户）
   - 收入信号（定价页面、计费代码、Stripe 集成）
   - 竞争护城河（哪些方面难以复制？）
4. 识别：能让这个项目增长 10 倍的那一件事
5. 标记：你正在构建但无关紧要的事情
```

### 模式 3：用户旅程审计

映射真实的用户体验：

```
1. 以新用户身份克隆/安装产品
2. 记录每一个摩擦点（令人困惑的步骤、错误、缺失的文档）
3. 为每个步骤计时
4. 与竞争对手的入门流程进行比较
5. 评分：价值实现时间（用户需要多久才能获得首次成功？）
6. 建议：优化入门流程的前3项改进措施
```

### 模式 4：功能优先级排序

当你有 10 个想法但需要选择 2 个时：

```
1. 列出所有候选功能
2. 为每项功能评分：影响力 (1-5) × 置信度 (1-5) ÷ 工作量 (1-5)
3. 按 ICE 分数排序
4. 应用约束条件：跑道、团队规模、依赖关系
5. 输出：带有理由的优先级路线图
```

## 输出

所有模式都输出可操作的文档，而非长篇大论。每条建议都有具体的后续步骤。

## 集成

可与以下工具配合使用：

* `/browser-qa` 用于验证用户旅程审计结果
* `/design-system audit` 用于视觉润色评估
* `/canary-watch` 用于发布后监控
