# 工作区表面审计

用于回答“这个工作区和机器现在实际能做什么，接下来我们应该添加或启用什么？”的只读审计技能。

这是 ECC 原生的 setup-audit 插件替代方案。除非用户明确要求后续实现，否则它不会修改文件。

## 何时使用

* 用户说“设置 Claude Code”、“推荐自动化”、“我应该使用什么插件或 MCP？”或“我缺少什么？”
* 在安装更多技能、钩子或连接器之前，审计机器或代码库
* 将官方市场插件与 ECC 原生覆盖范围进行比较
* 审查 `.env`、`.mcp.json`、插件设置或已连接应用表面，以查找缺失的工作流层
* 决定某个功能应该是技能、钩子、代理、MCP 还是外部连接器

## 不可协商的规则

* 绝不打印密钥值。仅展示提供商名称、功能名称、文件路径以及密钥或配置是否存在。
* 当 ECC 可以合理拥有该表面时，优先选择 ECC 原生工作流，而非通用的“安装另一个插件”建议。
* 将外部插件视为基准和灵感来源，而非权威的产品边界。
* 清晰区分三件事：
  * 现已可用
  * 可用但未在 ECC 中良好封装
  * 不可用，需要新的集成

## 审计输入

仅检查为良好回答问题所需的文件和设置：

1. 代码库表面
   * `package.json`、锁文件、语言标记、框架配置、`README.md`
   * `.mcp.json`、`.lsp.json`、`.claude/settings*.json`、`.codex/*`
   * `AGENTS.md`、`CLAUDE.md`、安装清单、钩子配置
2. 环境表面
   * 活动代码库及明显相邻 ECC 工作区中的 `.env*` 文件
   * 仅展示密钥名称，例如 `STRIPE_API_KEY`、`TWILIO_AUTH_TOKEN`、`FAL_KEY`
3. 已连接工具表面
   * 已安装的插件、启用的连接器、MCP 服务器、LSP 和应用集成
4. ECC 表面
   * 已覆盖需求的现有技能、命令、钩子、代理和安装模块

## 审计流程

### 阶段 1：盘点现有内容

生成简洁的清单：

* 活动的 harness 目标
* 已安装的插件和已连接的应用
* 已配置的 MCP 服务器
* 已配置的 LSP 服务器
* 由密钥名称暗示的环境支持服务
* 与工作区相关的现有 ECC 技能

如果某个表面仅作为原始功能存在，请明确指出。例如：

* “Stripe 可通过已连接应用使用，但 ECC 缺少 billing-operator 技能”
* “Google Drive 已连接，但没有 ECC 原生的 Google Workspace 操作员工作流”

### 阶段 2：与官方及已安装表面进行基准比较

将工作区与以下内容进行比较：

* 与设置、审查、文档、设计或工作流质量重叠的官方 Claude 插件
* Claude 或 Codex 中本地安装的插件
* 用户当前已连接的应用表面

不要仅仅列出名称。对于每个比较，回答：

1. 它们实际做什么
2. ECC 是否已具备同等功能
3. ECC 是否仅具备原始功能
4. ECC 是否完全缺失该工作流

### 阶段 3：将差距转化为 ECC 决策

对于每个真正的差距，推荐正确的 ECC 原生形态：

| 差距类型 | 首选的 ECC 形态 |
|----------|---------------------|
| 可重复的操作员工作流 | 技能 |
| 自动强制执行或副作用 | 钩子 |
| 专业化的委托角色 | 代理 |
| 外部工具桥接 | MCP 服务器或连接器 |
| 安装/引导指南 | 设置或审计技能 |

当需求是操作性的而非基础架构性的时，默认选择编排现有工具的面向用户的技能。

## 输出格式

按顺序返回五个部分：

1. **当前表面**
   * 现已可用的内容
2. **同等功能**
   * ECC 已匹配或超越基准的地方
3. **仅原始功能的差距**
   * 工具存在，但 ECC 缺少清晰的操作员技能
4. **缺失的集成**
   * 尚不可用的功能
5. **接下来 3-5 个最佳行动**
   * 具体的 ECC 原生添加项，按影响力排序

## 推荐规则

* 每个类别最多推荐 1-2 个最高价值的想法。
* 优先选择具有明显用户意图和业务价值的技能：
  * 设置审计
  * 计费/客户运营
  * 问题/项目运营
  * Google Workspace 运营
  * 部署/运营控制
* 如果连接器是公司特定的，仅在其真正可用或对用户工作流明显有用时才推荐。
* 如果 ECC 已具备强大的原始功能，建议封装技能，而非发明全新的子系统。

## 良好结果

* 用户可以立即看到已连接的内容、缺失的内容以及 ECC 接下来应拥有的内容。
* 推荐足够具体，无需另一次发现过程即可在代码库中实现。
* 最终答案围绕工作流组织，而非 API 品牌。
