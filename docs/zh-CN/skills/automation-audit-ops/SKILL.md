# 自动化审计运维

当用户询问哪些自动化正在运行、哪些作业已损坏、存在哪些重叠，或者当前哪些工具和连接器实际在做有用工作时，请使用此技能。

这是一个审计优先的操作员技能。其任务是在重写任何内容之前，生成一份有证据支持的清单，以及一套保留/合并/删除/下一步修复的建议。

## 技能栈

在相关时，将以下 ECC 原生技能纳入工作流：

* `workspace-surface-audit` 用于连接器、MCP、钩子和应用清单
* `knowledge-ops` 当审计需要协调实时仓库状态与持久化上下文时
* `github-ops` 当答案取决于 CI、计划工作流、问题或 PR 自动化时
* `ecc-tools-cost-audit` 当真正的问题是 webhook 扇出、排队作业或兄弟应用仓库中的计费消耗时
* `research-ops` 当必须将本地清单与当前平台支持或公共文档进行比较时
* `verification-loop` 用于证明修复后的状态，而非依赖假设的恢复

## 使用时机

* 用户询问"我有哪些自动化"、"哪些正在运行"、"哪些已损坏"或"哪些有重叠"
* 任务涉及 cron 作业、GitHub Actions、本地钩子、MCP 服务器、连接器、包装器或应用集成
* 用户想知道哪些是从其他代理系统移植过来的，哪些仍需要在 ECC 内部重建
* 工作区积累了多种实现相同功能的方式，用户希望有一个规范路径

## 防护措施

* 除非用户明确要求修复，否则以只读模式开始
* 区分：
  * 已配置
  * 已认证
  * 最近已验证
  * 陈旧或损坏
  * 完全缺失
* 不要仅因为某个技能或配置引用了某个工具，就声称该工具正在运行
* 在证据表存在之前，不要合并或删除重叠的表面

## 工作流

### 1. 盘点实际表面

在理论推测之前，先读取当前的实时表面：

* 仓库钩子和本地钩子脚本
* GitHub Actions 和计划工作流
* MCP 配置和启用的服务器
* 连接器或应用支持的集成
* 包装器脚本和仓库特定的自动化入口点

按表面分组：

* 本地运行时
* 仓库 CI / 自动化
* 连接的外部系统
* 消息传递 / 通知
* 计费 / 客户运营
* 研究 / 监控

### 2. 按运行状态对每个项目进行分类

对于每个已发现的自动化，标记：

* 已配置
* 已认证
* 最近已验证
* 陈旧或损坏
* 缺失

然后对问题类型进行分类：

* 活动中断
* 认证故障
* 状态陈旧
* 重叠或冗余
* 功能缺失

### 3. 追踪证明路径

为每个重要声明提供具体来源支持：

* 文件路径
* 工作流运行
* 钩子日志
* 配置条目
* 最近的命令输出
* 确切的故障特征

如果当前状态不明确，请直接说明，不要假装审计已完成。

### 4. 以保留/合并/删除/下一步修复结束

对于每个重叠或可疑的表面，返回一个调用：

* 保留
* 合并
* 删除
* 下一步修复

其价值在于将嘈杂的自动化整合到一个规范的 ECC 路径中，而不是保留每一条历史路径。

## 输出格式

```text
当前表面
- 自动化
- 来源
- 实时状态
- 证明

发现
- 活动中断
- 重叠
- 陈旧状态
- 缺失能力

建议
- 保留
- 合并
- 削减
- 下次修复

下一步 ECC 行动
- 确切的技能 / 钩子 / 工作流 / 应用通道以加强
```

## 陷阱

* 当可以读取实时清单时，不要凭记忆回答
* 不要将"存在于配置中"视为"正常工作"
* 在指出已损坏的高价值路径之前，不要修复低价值的冗余
* 如果用户首先要求的是清单，不要将任务扩展为仓库重写

## 验证

* 重要声明引用了实时证明路径
* 每个已发现的自动化都标有清晰的运行状态类别
* 最终建议区分了保留/合并/删除/下一步修复
