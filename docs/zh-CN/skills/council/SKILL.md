# 顾问团

为模糊决策召集四位顾问：

* 上下文中的 Claude 声音
* 一位怀疑论者子代理
* 一位实用主义者子代理
* 一位批评者子代理

这适用于**模糊情况下的决策**，而非代码审查、实施计划或架构设计。

## 何时使用

在以下情况使用顾问团：

* 一个决策有多个可行路径且没有明显的赢家
* 需要明确地呈现权衡取舍
* 用户要求第二意见、异议或多重视角
* 对话锚定是一个真实风险
* 一个“执行/不执行”的决策将受益于对抗性质疑

示例：

* 单体仓库 vs 多仓库
* 立即发布 vs 等待打磨
* 功能开关 vs 全面推出
* 简化范围 vs 保持战略广度

## 何时不使用

| 代替顾问团 | 使用 |
| --- | --- |
| 验证输出是否正确 | `santa-method` |
| 将功能分解为实施步骤 | `planner` |
| 设计系统架构 | `architect` |
| 审查代码中的错误或安全问题 | `code-reviewer` 或 `santa-method` |
| 直接的事实性问题 | 直接回答即可 |
| 明确的执行任务 | 直接执行任务即可 |

## 角色

| 声音 | 视角 |
| --- | --- |
| 架构师 | 正确性、可维护性、长期影响 |
| 怀疑论者 | 挑战前提、简化、打破假设 |
| 实用主义者 | 发布速度、用户影响、操作现实 |
| 批评者 | 边缘情况、下行风险、失败模式 |

三个外部声音应作为全新的子代理启动，**仅包含问题和相关上下文**，而非完整的持续对话。这是反锚定机制。

## 工作流程

### 1. 提取真实问题

将决策简化为一个明确的提示：

* 我们要决定什么？
* 哪些约束条件重要？
* 什么算作成功？

如果问题模糊，在召集顾问团之前提出一个澄清性问题。

### 2. 仅收集必要的上下文

如果决策与代码库相关：

* 收集相关文件、代码片段、问题文本或指标
* 保持简洁
* 仅包含做出决策所需的上下文

如果决策是战略性的/通用的：

* 除非代码库片段实质性改变答案，否则跳过它们

### 3. 首先形成架构师立场

在阅读其他声音之前，写下：

* 你的初始立场
* 支持该立场的三个最强理由
* 你首选路径中的主要风险

首先完成这一步，以便综合过程不会简单地反映外部声音。

### 4. 并行启动三个独立的声音

每个子代理获得：

* 决策问题
* 简洁的上下文（如果需要）
* 严格的角色
* 不必要的对话历史

提示格式：

```text
你是四人决策委员会中的[角色]。

问题：
[决策问题]

背景：
[仅相关片段或约束条件]

请按以下格式回应：
1. 立场 — 1-2句话
2. 推理 — 3个简洁要点
3. 风险 — 你建议中的最大风险
4. 意外因素 — 其他成员可能忽略的一点

请直接回应。不要模棱两可。总字数控制在300字以内。
```

角色强调：

* 怀疑论者：挑战框架，质疑假设，提出最简单可信的替代方案
* 实用主义者：优化速度、简单性和现实世界执行
* 批评者：揭示下行风险、边缘情况以及计划可能失败的原因

### 5. 使用偏见防护进行综合

你既是参与者又是综合者，因此请使用以下规则：

* 不要在不解释原因的情况下驳回外部观点
* 如果外部声音改变了你的建议，请明确说明
* 即使你拒绝，也要始终包含最强的异议
* 如果两个声音联合反对你的初始立场，将其视为一个真实信号
* 在裁决前保持原始立场可见

### 6. 呈现简洁的裁决

使用此输出格式：

```markdown
## 委员会：[简短决策标题]

**架构师：**[1-2 句立场]
[1 行理由]

**质疑者：**[1-2 句立场]
[1 行理由]

**务实派：**[1-2 句立场]
[1 行理由]

**批评者：**[1-2 句立场]
[1 行理由]

### 裁决
- **共识：**[他们达成一致的地方]
- **最强异议：**[最重要的分歧]
- **前提检查：**[质疑者是否挑战了问题本身？]
- **建议：**[综合得出的路径]
```

使其在手机屏幕上易于浏览。

## 持久性规则

**不要**将临时笔记写入 `~/.claude/notes` 或此技能的其他影子路径。

如果顾问团实质性改变了建议：

* 使用 `knowledge-ops` 将经验教训存储在正确的持久位置
* 或者，如果结果属于会话记忆，则使用 `/save-session`
* 或者，如果决策改变了正在执行的实际情况，则直接更新相关的 GitHub / Linear 问题

仅在决策改变某些真实情况时才持久化决策。

## 多轮后续跟进

默认为一轮。

如果用户想要另一轮：

* 保持新问题聚焦
* 仅在必要时包含之前的裁决
* 尽可能保持怀疑论者的独立性，以保留反锚定价值

## 反模式

* 使用顾问团进行代码审查
* 在任务仅仅是实施工作时使用顾问团
* 向子代理提供整个对话记录
* 在最终裁决中隐藏分歧
* 无论重要性如何，都将每个决策作为笔记持久化

## 相关技能

* `santa-method` — 对抗性验证
* `knowledge-ops` — 正确持久化决策差异
* `search-first` — 在召集顾问团之前收集外部参考资料（如果需要）
* `architecture-decision-records` — 当决策成为长期系统策略时，将结果正式化

## 示例

问题：

```text
我们是现在就将 ECC 2.0 作为 alpha 版本发布，还是等到控制平面 UI 更完善后再发布？
```

可能的顾问团形态：

* 架构师推动结构完整性并避免混乱的表面
* 怀疑论者质疑 UI 是否真的是瓶颈因素
* 实用主义者询问现在可以发布什么而不损害信任
* 批评者关注支持负担、期望债务和推出混乱

价值不在于一致同意。价值在于在选择之前使分歧清晰可见。
