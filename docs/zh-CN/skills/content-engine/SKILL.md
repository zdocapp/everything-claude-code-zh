# 内容引擎

在不将作者真实声音扁平化为平台口水话的前提下，构建平台原生内容。

## 何时启用

* 撰写 X 帖子或推文串时
* 起草 LinkedIn 帖子或发布更新时
* 为短视频或 YouTube 解说撰写脚本时
* 将文章、播客、演示、文档或内部笔记转化为公开内容时
* 围绕产品、见解或叙事构建发布序列或持续内容体系时

## 不可妥协项

1. 从源材料开始，而非通用帖子公式。
2. 为平台调整格式，而非调整人物形象。
3. 一个帖子应承载一个实际主张。
4. 具体性胜过形容词。
5. 除非用户明确要求，否则不使用互动诱饵。

## 源材料优先工作流

起草前，先确定源材料集：

* 已发表的文章
* 笔记或内部备忘录
* 产品演示
* 文档或更新日志
* 文字记录
* 截图
* 同一作者之前的帖子

如果用户想要特定声音，请在写作前根据真实示例构建声音档案。
当声音一致性对多个输出很重要时，使用 `brand-voice` 作为规范工作流。

## 声音处理

`brand-voice` 是规范的声音层。

在以下情况下首先运行它：

* 有多个下游输出时
* 用户明确关心写作风格时
* 内容涉及发布、推广或声誉敏感时

在此处复用生成的 `VOICE PROFILE`，而不是重建第二个声音模型。
如果用户特别想要 Affaan / ECC 声音，仍应将 `brand-voice` 视为事实来源，并向其提供可用的最佳实时或源自源材料的素材。

## 硬性禁令

删除并重写以下任何内容：

* "在当今快速发展的格局中"
* "颠覆者"、"革命性的"、"前沿的"
* "这为什么重要"，除非后面紧跟具体内容
* 仅为了获取回复而以 LinkedIn 风格的问题结尾
* 在 LinkedIn 上强装随意
* 源材料中不存在的虚假互动填充内容

## 平台适配规则

### X

* 以最有力的主张、成果或矛盾点开头
* 如果源声音是压缩的，保持其压缩感
* 如果撰写推文串，每条推文都必须推进论点
* 不要填充受众不需要的背景信息

### LinkedIn

* 仅扩展至让非直接利基领域的人能理解的程度
* 除非源材料本身具有反思性，否则不要将其变成虚假的"经验教训"帖
* 不使用企业激励语调
* 不堆砌赞美，不使用"旅程"等填充词

### 短视频

* 围绕视觉序列和证明点撰写脚本
* 前几秒应展示结果、问题或亮点
* 不要撰写纸上谈兵优于屏幕呈现的旁白

### YouTube

* 尽早展示结果或矛盾点
* 按论点或进展组织，而非填充性章节
* 仅在有助于清晰度时使用章节划分

### 新闻简报

* 以要点、冲突或成果开头
* 不要在首段进行预热
* 每个部分都需要增加新内容

## 内容再利用流程

1. 选择核心资产。
2. 提取 3 到 7 个原子化主张或场景。
3. 根据尖锐性、新颖性和证明力进行排序。
4. 为每个输出分配一个强有力的想法。
5. 为每个平台调整结构。
6. 剥离平台化的填充内容。
7. 运行质量检查。

## 交付物

当被要求提供活动方案时，返回：

* 如果声音匹配很重要，则提供简短的声音档案
* 核心角度
* 平台原生草稿
* 仅在有助于执行时提供发布顺序
* 发布前必须填补的空白

## 质量检查

交付前确保：

* 每个草稿听起来都像目标作者，而非平台刻板印象
* 每个草稿都包含一个真实的主张、证明点或具体观察
* 没有留下通用的炒作语言
* 没有留下虚假的互动诱饵
* 除非要求，否则跨平台没有重复的文案
* 任何行动号召都是应得的且经过用户批准的

## 相关技能

* `brand-voice` 用于源自源材料的声音档案
* `crosspost` 用于平台特定分发
* `x-api` 用于获取近期帖子和发布已批准的 X 输出
