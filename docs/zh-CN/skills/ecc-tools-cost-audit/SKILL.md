# ECC 工具成本审计

当用户怀疑 ECC Tools GitHub 应用正在消耗成本、过度创建 PR、绕过使用限制或将免费用户路由到高级分析路径时，使用此技能。

这是针对兄弟仓库 [ECC-Tools](../../../../ECC-Tools) 的聚焦操作员工作流。它不是通用的计费技能，也不是仓库范围的代码审查。

## 技能栈

在相关时将以下 ECC 原生技能拉入工作流：

* `autonomous-loops` 用于跨越 webhook、队列、计费和重试的有界多步骤审计
* `agentic-engineering` 用于将请求路径追踪到离散、可证明的单元
* `customer-billing-ops` 当需要清晰分离仓库行为和客户影响计算时
* `search-first` 在发明辅助工具或重新实现仓库本地实用程序之前
* `security-review` 当涉及身份验证、使用限制、授权或许可时
* `verification-loop` 用于证明重新运行的安全性和修复后的确切状态
* `tdd-workflow` 当修复需要在 worker、路由器或计费路径中进行回归覆盖时

## 何时使用

* 用户提到 ECC Tools 消耗率、PR 递归、过度创建的 PR、使用限制绕过或高级模型泄漏
* 任务在兄弟仓库 `ECC-Tools` 中，并且依赖于 webhook 处理器、队列 worker、使用量预留、PR 创建逻辑或付费网关强制执行
* 客户报告称应用创建了过多 PR、计费不正确，或分析了代码但未产生可用结果

## 范围护栏

* 在兄弟仓库 `ECC-Tools` 中工作，而不是在 `everything-claude-code` 中
* 除非用户明确要求修复，否则以只读模式开始
* 在追踪分析消耗时，不要修改无关的计费、结账或 UI 流程
* 在证明其合理性之前，将应用生成的分支和应用生成的 PR 视为危险递归路径
* 明确区分以下三件事：
  * 仓库侧的消耗根本原因
  * 面向客户的计费影响
  * 需要后续产品待办事项跟进的产品或授权差距

## 工作流

### 1. 冻结仓库范围

* 切换到兄弟仓库 `ECC-Tools`
* 首先检查分支和本地差异
* 确定审计的确切范围：
  * webhook 路由器
  * 队列生产者
  * 队列消费者
  * PR 创建路径
  * 使用量预留 / 计费路径
  * 模型路由路径

### 2. 在理论化之前追踪入口

* 首先检查 `src/index.*` 或主要入口点
* 在建议修复之前，映射每个入队路径
* 确认哪些 GitHub 事件共享队列类型
* 确认推送、pull\_request、同步、评论或手动重新运行事件是否可能汇聚到相同的高成本路径

### 3. 追踪 worker 和副作用

* 检查处理分析的队列消费者或计划 worker
* 确认排队的分析是否总是以以下方式结束：
  * PR 创建
  * 分支创建
  * 文件更新
  * 高级模型调用
  * 使用量增加
* 如果分析可以在输出持久化之前消耗令牌然后失败，则将其归类为"消耗但输出损坏"

### 4. 审计高信号消耗路径

#### PR 倍增

* 检查 PR 辅助函数和分支命名
* 检查去重、同步事件处理和现有 PR 重用
* 如果应用生成的分支可以重新进入分析，则将其视为优先级为 0 的递归风险

#### 配额绕过

* 检查配额检查的位置与使用量预留或增加的位置
* 如果在入队前检查配额，但仅在 worker 内部计费使用量，则将并发的前门通过视为真实的竞争条件

#### 高级模型泄漏

* 检查模型选择、层级分支和提供商路由
* 验证免费或受限制的用户在存在高级密钥时是否仍能访问高级分析器

#### 重试消耗

* 检查重试循环、重复队列作业和确定性失败重新运行
* 如果相同的非暂时性错误可以重复消耗分析，则在质量改进之前修复此问题

### 5. 按消耗顺序修复

如果用户要求进行代码更改，请按以下顺序优先修复：

1. 停止自动 PR 倍增
2. 停止配额绕过
3. 停止高级模型泄漏
4. 停止重复作业扇出和无意义的重试
5. 关闭重新运行/更新的安全漏洞

除非同一根本原因明显跨越多个文件，否则将此次修复限制在一到三个直接修复内。

### 6. 用最小的验证步骤进行验证

* 仅重新运行覆盖更改路径的针对性测试或集成切片
* 验证消耗路径现在是否：
  * 被阻止
  * 已去重
  * 降级为更便宜的分析
  * 或早期被拒绝
* 准确说明最终状态：
  * 本地已更改
  * 本地已验证
  * 已推送
  * 已部署
  * 仍被阻止

## 高信号失败模式

### 1. 所有触发器使用一种队列类型

如果推送、PR 同步和手动审计都将同一作业入队，并且 worker 总是创建 PR，那么分析就等于 PR 垃圾邮件。

### 2. 入队后使用量预留

如果使用量在前门检查，但仅在 worker 内部增加，则并发请求可能全部通过网关并超出配额。

### 3. 免费层级进入高级路径

如果排队的免费作业在密钥存在时仍能路由到 Anthropic 或其他高级提供商，那么即使用户从未看到高级结果，这也是真实的支出泄漏。

### 4. 应用生成的分支重新进入 webhook

如果 `pull_request.synchronize`、分支推送或评论触发的运行在应用拥有的分支上触发，则应用可以递归分析其自己的输出。

### 5. 持久化安全之前的高成本工作

如果系统可以在 PR 创建、文件更新或分支冲突时消耗令牌然后失败，那么它就是在消耗成本而没有交付价值。

## 陷阱

* 不要从广泛的仓库漫游开始；首先确定 webhook -> 队列 -> worker
* 不要将客户计费推断与基于代码的产品事实混为一谈
* 在最高消耗路径得到控制之前，不要修复低价值的质量问题
* 在狭窄的验证步骤重新运行之前，不要声称消耗已修复
* 除非用户要求，否则不要推送或部署
* 如果无关的仓库本地更改已在进行中，请不要触碰

## 验证

* 根本原因引用确切的文件路径和代码区域
* 修复按消耗影响排序，而不是代码整洁度
* 验证命令已命名
* 最终状态区分本地更改、验证、推送和部署
