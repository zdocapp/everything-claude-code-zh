# 产品能力

此技能将产品意图转化为明确的工程约束。

当差距不在于“我们应该构建什么？”而在于“在实施开始之前，究竟必须满足哪些条件？”时使用它。

## 何时使用

* 存在 PRD、路线图项目、讨论或创始人笔记，但实施约束仍然隐含
* 某个功能涉及多个服务、代码库或团队，需要在编码前制定能力契约
* 产品意图明确，但架构、数据、生命周期或策略影响仍然模糊
* 高级工程师在评审期间不断重申相同的隐藏假设
* 您需要一个可跨 harness 和会话复用的工件

## 规范工件

如果代码库有持久的产品上下文文件，例如 `PRODUCT.md`、`docs/product/` 或程序规范目录，请在那里更新。

如果尚不存在能力清单，请使用以下模板创建：

* `docs/examples/product-capability-template.md`

目标不是创建另一个规划栈。目标是使隐藏的能力约束持久化且可复用。

## 不可协商的规则

* 不要编造产品事实。明确标记未解决的问题。
* 将用户可见的承诺与实现细节分开。
* 明确指出哪些是固定策略，哪些是架构偏好，哪些仍然开放。
* 如果请求与现有代码库约束冲突，请清楚说明，而不是含糊带过。
* 优先使用一个可复用的能力工件，而不是分散的临时笔记。

## 输入

仅读取所需内容：

1. 产品意图
   * issue、讨论、PRD、路线图笔记、创始人消息
2. 当前架构
   * 相关的代码库文档、契约、模式、路由、现有工作流
3. 现有能力上下文
   * `PRODUCT.md`、设计文档、RFC、迁移笔记、运营模型文档
4. 交付约束
   * 认证、计费、合规、发布、向后兼容性、性能、评审策略

## 核心工作流

### 1. 重述能力

将需求压缩成一个精确的陈述：

* 用户或操作者是谁
* 此功能上线后存在什么新能力
* 因为它，什么结果发生了变化

如果此陈述薄弱，实施将会偏离。

### 2. 解析能力约束

提取在实施前必须满足的约束：

* 业务规则
* 范围边界
* 不变量
* 信任边界
* 数据所有权
* 生命周期转换
* 发布 / 迁移要求
* 故障和恢复预期

这些通常是仅存在于高级工程师记忆中的东西。

### 3. 定义面向实现的契约

生成一个 SRS 风格的能力计划，包含：

* 能力摘要
* 明确的非目标
* 参与者和界面
* 所需状态和转换
* 接口 / 输入 / 输出
* 数据模型影响
* 安全 / 计费 / 策略约束
* 可观测性和操作员要求
* 阻碍实施的开放性问题

### 4. 转化为执行

以确切的交接结束：

* 已准备好直接实施
* 需要先进行架构评审
* 需要先进行产品澄清

如果有用，指向下一个 ECC 原生通道：

* `project-flow-ops`
* `workspace-surface-audit`
* `api-connector-builder`
* `dashboard-builder`
* `tdd-workflow`
* `verification-loop`

## 输出格式

按此顺序返回结果：

```text
能力
- 一段式重述

约束条件
- 固定规则、不变条件和边界

实施契约
- 参与者
- 交互界面
- 状态与转换
- 接口/数据影响

非目标
- 本工作流明确不负责的事项

开放性问题
- 阻碍因素或仍需做出的产品决策

交接事项
- 后续步骤及应由哪个ECC工作流接手
```

## 良好结果

* 产品意图现在足够具体，可以在不中途重新发现隐藏约束的情况下实施。
* 工程评审拥有持久的工件，而不是依赖记忆或 Slack 上下文。
* 生成的计划可在 Claude Code、Codex、Cursor、OpenCode 和 ECC 2.0 规划界面中复用。
