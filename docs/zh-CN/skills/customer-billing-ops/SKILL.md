# 客户账单运营

此技能用于真实的客户运营，而非通用的支付 API 设计。

目标是帮助运营人员回答：这位客户是谁，发生了什么，最安全的修复方案是什么，以及我们应该发送什么后续跟进？

## 何时使用

* 客户表示账单出错、要求退款或无法取消
* 调查重复订阅、意外扣款、续订失败或流失风险
* 审查套餐组合、有效订阅、年度与月度转换情况或团队席位混淆
* 创建或验证账单门户流程
* 审计涉及订阅、发票、退款或支付方式的客服投诉

## 首选工具界面

* 优先使用已连接的账单工具，例如 Stripe
* 仅将电子邮件、GitHub 或问题跟踪器作为辅助证据
* 当平台已提供所需控制功能时，优先选择托管的账单/客户门户，而非自定义账户管理代码

## 防护措施

* 切勿在回复中暴露密钥、完整卡片详情或不必要的客户个人身份信息
* 不要盲目退款；首先对问题进行归类
* 区分以下情况：
  * 意外重复购买
  * 有意的多席位或团队购买
  * 产品故障 / 价值未实现
  * 支付失败或结账未完成
  * 因缺少自助服务控制而取消
* 对于年度计划、团队计划和按比例计费状态，在采取行动前先核实合同结构

## 工作流程

### 1. 清晰地识别客户

从可用的最强标识符开始：

* 客户邮箱
* Stripe 客户 ID
* 订阅 ID
* 发票 ID
* GitHub 用户名或客服邮箱（如果已知其与账单关联）

返回简洁的身份摘要：

* 客户
* 有效订阅
* 已取消的订阅
* 发票
* 明显的异常情况，例如重复的有效订阅

### 2. 对问题进行归类

在采取行动前，将案例归入以下类别之一：

| 案例 | 典型操作 |
|------|----------------|
| 重复的个人订阅 | 取消额外订阅，考虑退款 |
| 真实的多席位/团队意图 | 保留席位，澄清计费模式 |
| 支付失败 / 结账未完成 | 通过门户恢复或更新支付方式 |
| 缺少自助服务控制 | 提供门户、取消路径或发票访问权限 |
| 产品故障或信任破裂 | 退款、致歉、记录产品问题 |

### 3. 首先采取最安全且可逆的操作

优先顺序：

1. 恢复自助服务管理
2. 修复重复或错误的账单状态
3. 仅对受影响的费用或重复项退款
4. 记录原因
5. 发送简短的客户跟进信息

如果修复需要产品工作，则分开处理：

* 立即进行客户补救
* 产品缺陷 / 工作流程漏洞纳入待办事项

### 4. 检查运营端的产品漏洞

如果客户痛点源于缺少运营界面，请明确指出。常见示例：

* 没有账单门户
* 没有使用量/速率限制可见性
* 没有套餐/席位说明
* 没有取消流程
* 没有重复订阅防护

将这些视为 ECC 或网站后续跟进事项，而不仅仅是客服事件。

### 5. 生成运营交接

以以下内容结束：

* 客户状态摘要
* 已采取的操作
* 收入影响
* 待发送的跟进文本
* 待创建的产品或待办事项问题

## 输出格式

使用此结构：

```text
客户
- 姓名 / 邮箱
- 相关账户标识符

账单状态
- 有效订阅
- 发票或续订状态
- 异常情况

决策
- 问题分类
- 为何此操作正确

采取的行动
- 退款 / 取消 / 门户 / 无操作

后续跟进
- 简短的客户消息

产品差距
- 应在产品或网站中修复的问题
```

## 良好建议示例

* "正确的解决方案是账单门户，而不是自定义仪表板"
* "这看起来像是重复的个人结账，而非真实的团队席位购买"
* "退还一笔重复扣款，保留剩余的有效订阅，然后根据需要稍后将客户转换为组织计费"
