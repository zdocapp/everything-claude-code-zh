# 基准测试 — 性能基线 & 回归检测

## 使用时机

* 在 PR 前后，用于衡量性能影响
* 为项目建立性能基线
* 当用户报告“感觉变慢”时
* 在发布前 — 确保达到性能目标
* 将你的技术栈与替代方案进行比较

## 工作原理

### 模式 1：页面性能

通过浏览器 MCP 测量真实的浏览器指标：

```
1. 访问每个目标 URL
2. 测量核心 Web 指标：
   - LCP (最大内容绘制) — 目标 < 2.5 秒
   - CLS (累积布局偏移) — 目标 < 0.1
   - INP (下一次绘制交互) — 目标 < 200 毫秒
   - FCP (首次内容绘制) — 目标 < 1.8 秒
   - TTFB (首字节时间) — 目标 < 800 毫秒
3. 测量资源大小：
   - 总页面大小 (目标 < 1MB)
   - JS 包大小 (目标 < 200KB gzipped)
   - CSS 大小
   - 图片大小
   - 第三方脚本大小
4. 统计网络请求数量
5. 检查是否存在渲染阻塞资源
```

### 模式 2：API 性能

对 API 端点进行基准测试：

```
1. 对每个端点进行 100 次请求
2. 测量：p50、p95、p99 延迟
3. 跟踪：响应大小、状态码
4. 负载测试：10 个并发请求
5. 与 SLA 目标进行对比
```

### 模式 3：构建性能

测量开发反馈循环：

```
1. 冷构建时间
2. 热重载时间 (HMR)
3. 测试套件持续时间
4. TypeScript 检查时间
5. 代码检查时间
6. Docker 构建时间
```

### 模式 4：前后对比

在变更前后运行以衡量影响：

```
/benchmark baseline    # 保存当前指标
# ... 进行更改 ...
/benchmark compare     # 与基线进行比较
```

输出：

```
| Metric | Before | After | Delta | Verdict |
|--------|--------|-------|-------|---------|
| LCP | 1.2s | 1.4s | +200ms | WARNING: WARN |
| Bundle | 180KB | 175KB | -5KB | ✓ BETTER |
| Build | 12s | 14s | +2s | WARNING: WARN |
```

## 输出

将基线以 JSON 格式存储在 `.ecc/benchmarks/` 中。通过 Git 跟踪，以便团队共享基线。

## 集成

* CI：在每个 PR 上运行 `/benchmark compare`
* 与 `/canary-watch` 搭配使用，用于部署后监控
* 与 `/browser-qa` 搭配使用，用于完整的发布前检查清单
