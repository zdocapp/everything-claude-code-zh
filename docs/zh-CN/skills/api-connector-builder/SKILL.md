# API 连接器构建器

当任务是添加一个与代码库原生集成的接口，而不仅仅是通用的 HTTP 客户端时，请使用此方法。

关键在于匹配宿主代码库的模式：

* 连接器布局
* 配置模式
* 认证模型
* 错误处理
* 测试风格
* 注册/发现机制

## 使用时机

* “为此项目构建一个 Jira 连接器”
* “按照现有模式添加一个 Slack 提供程序”
* “为此 API 创建一个新的集成”
* “构建一个匹配代码库连接器风格的插件”

## 约束准则

* 如果代码库已有集成架构，不要发明新的
* 不要仅从供应商文档开始；首先从代码库内现有的连接器开始
* 如果代码库期望包含注册机制、测试和文档，不要只停留在传输代码层面
* 如果代码库有更新的当前模式，不要盲目照搬旧的连接器

## 工作流程

### 1. 了解项目风格

检查至少 2 个现有的连接器/提供程序，并梳理：

* 文件布局
* 抽象边界
* 配置模型
* 重试/分页约定
* 注册钩子
* 测试夹具和命名

### 2. 明确目标集成范围

仅定义代码库实际需要的接口：

* 认证流程
* 关键实体
* 核心读写操作
* 分页和速率限制
* Webhook 或轮询模型

### 3. 以代码库原生层次构建

典型分层：

* 配置/模式
* 客户端/传输层
* 映射层
* 连接器/提供程序入口点
* 注册
* 测试

### 4. 对照源模式进行验证

新的连接器在代码库中应该看起来是自然的，而不是从不同生态系统导入的。

## 参考形态

### 提供程序风格

```text
providers/
  existing_provider/
    __init__.py
    provider.py
    config.py
```

### 连接器风格

```text
integrations/
  existing/
    client.py
    models.py
    connector.py
```

### TypeScript 插件风格

```text
src/integrations/
  existing/
    index.ts
    client.ts
    types.ts
    test.ts
```

* \[ ] 测试镜像了宿主代码库的风格
